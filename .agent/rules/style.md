ЕДИНЫЕ ПРАВИЛА UI/UX И СТИЛИСТИКИ ПРОЕКТА

1. Общий принцип

При создании любой новой страницы, компонента или UI-блока необходимо строго придерживаться существующей дизайн-системы проекта.
Никакой «самодеятельности», даже если «так красивее».

2. Анализ перед реализацией (обязательно)

Перед созданием нового элемента:

Найти аналогичные компоненты или страницы в проекте
(layout, карточки, формы, модальные окна, списки, кнопки).

Проанализировать:

структуру

размеры

отступы

цветовую схему

типографику

hover / focus / active состояния

анимации и transition

Использовать их как эталон.

❗ Если подходящего примера нет — использовать ближайший по смыслу компонент, а не изобретать новый стиль.

3. Типографика

Использовать только шрифты, уже применяемые в проекте

Размеры текста:

Заголовки — строго по существующим уровням (h1–h6 или их аналоги)

Текст — без произвольных размеров

Запрещено:

Добавлять новые шрифты

Использовать inline-размеры (style="font-size")

4. Цвета

Использовать только цвета из дизайн-палитры проекта

Цвета должны применяться через:

CSS variables

Tailwind tokens

общие классы

Запрещено:

Хардкодить HEX / RGB без необходимости

Добавлять «чуть темнее / чуть светлее» от себя

5. Размеры и сетка

Все блоки должны:

соответствовать общей сетке

иметь одинаковые padding / margin с похожими блоками

Карточки, формы, таблицы:

одинаковая ширина

одинаковые скругления

одинаковая высота элементов управления

UI должен выглядеть так, будто его делал один человек в трезвом уме, а не 5 дизайнеров по очереди.

6. Компоненты

Перед созданием нового компонента:

проверить, нет ли уже существующего

Переиспользовать:

кнопки

инпуты

селекты

модалки

уведомления

Если компонент создаётся:

он должен быть универсальным

без жёсткой привязки к конкретной странице

7. Анимации и переходы

Использовать одинаковые transition:

длительность

easing

Анимации должны:

быть минимальными

не отвлекать от контента

Запрещено:

разные скорости «на глаз»

резкие анимации без причины

8. Именование и структура

Названия компонентов и классов:

логичные

предсказуемые

без «test», «new», «final2»

Структура файлов должна повторять уже принятую архитектуру проекта.

9. Проверка перед коммитом

Перед завершением работы:

Сравнить новый элемент с похожими страницами

Убедиться, что:

стилистика совпадает

нет визуальных выбросов

UI выглядит цельно

Если элемент «выделяется» — это баг, а не фича.

10. Главное правило

Если есть сомнения — копируй существующее поведение и внешний вид.
Консистентность важнее креативности.
